Handling Browser Refresh During Script Execution

Currently, the frontend makes a single HTTP POST request to /api/scripts/execute and waits for it to finish. If the browser is refreshed, the connection drops, but the Node.js server keeps running the Python script in the background until it finishes, while the generated result is lost.

There are two main approaches to solving this, depending on the user experience you want:

Option 1: Stop Execution on Disconnect (Easiest & Fastest)

If the user refreshes or closes the tab, we detect the dropped connection and immediately terminate the running Python script.

Pros: * Very easy to implement. * Prevents the server from wasting resources on background tasks no one is waiting for. * Prevents simultaneous runs from corrupting data if the user refreshes and clicks "Run" again.

Cons: * If the user accidentally refreshes a script that has been running for 5 minutes, they lose all their progress and have to start over.

Implementation: * Add a listener to the request object (req.on('close', ...)) in api.js. * If the connection closes before the response is explicitly sent, call .kill() on the generated Python child process.

Phase 1 (Current Goal): Simple UI Lock & History

This is the simplest way to protect the server load and give users visibility, without needing a complex Queue or polling system.

How it works (The Rules): 1. Environment-Level Execution Lock: Since multiple tenants share the same database within a specific Environment (e.g., all QA tenants hit the QA DB, all Prod tenants hit the Prod DB), the execution lock must be tracked per Environment (e.g., isExecuting['qa'] = true). 2. Execute Button Lockout & Race Conditions: * The "Download Template" button stays active so users can prepare data. * Only the final "Start Execution" (Upload & Run) button checks this lock. * Race Condition Protection: If User A and User B both have the modal open, and User A clicks "Execute", the server locks it. When User B clicks "Execute" a second later, the API immediately rejects User B's request with a "423 Locked" or "429 Too Many Requests" error. The frontend catches this error, stops the upload progress, and displays the warning: "An execution is already running in this environment. Please check History." 3. Synchronous Execution (Status Quo retained): The user who successfully started the script must keep their browser window open to get their final result file downloaded. 4. Resilience (History Only): If the user who triggered the run closes their browser, the backend script continues running to the end. Since they can't download the result, the system simply records the final outcome (pass/fail) in the History log.

Implementation Steps: * Backend (api.js): * Add a global variable const isExecuting = false; (or an object mapping tenantId: true). * Before starting a Python script in /api/scripts/execute, check this flag. If true, return 429 Too Many Requests (or similar). If false, set to true. * When the script completes or crashes, set it back to false. * Add logic to write a record to backend/history.json containing: scriptName, startTime, endTime, status, totalRecords, passCount, failCount, tenant. * Create a simple /api/scripts/history endpoint to read this JSON file. * Frontend (script_management_v2.js / HTML): * Before showing the upload modal (or via a quick /api/scripts/status check), verify if the execution lock is active. If so, disable the button and show the warning message. * Build the "History" view that simply displays the JSON records in a table. It does not need "Download Result" buttons, as results are ephemeral to the user's active session.

Option 2: Background Tasks, Polling, History, & Queuing (Advanced, Phase 2)

Instead of executing the script immediately and making the frontend wait, we move to a Job Queue System.

Pros: * Server & DB Protection: By enforcing a strict queue (e.g., only 1 or 2 scripts running globally at a time), we guarantee that a sudden spike in users won't crash the server or overload the database. * Resilient UX: If the user refreshes the page or logs out, their task continues in the queue or background. * Full Audit History: We keep a log of all past runs with detailed metrics so users can review the outcomes later.

Cons: * Requires significant refactoring of both the Backend (api.js) and the Frontend (script_management_v2.js & a new HTML view). * More complex to debug since tasks run in the background.

Implementation Breakdown:

1. The Queue Manager (queueManager.js) * To keep concerns strictly separated and prevent api.js from becoming bloated, we will create a dedicated backend/queueManager.js. * When /api/scripts/execute is called in api.js, it will simply pass the request data to the Queue Manager. * The Queue Manager will maintain the in-memory queue array, backed up by a persistent tasks.json file. * It will run the Task Worker background loop: - If active_tasks < MAX_CONCURRENT_TASKS, pull from queue, update state to "In Progress", and spawn the Python process. - When the Python script finishes/fails, handle saving outputs, update status to "Completed/Failed", and trigger the next task.

2. File Storage (Inputs & Outputs) * The Storage Challenge: Data Generation inputs and outputs can be large. * Current Initial Approach (Local Disk): - When a user uploads rows, we will convert the JSON rows back to a temporary .xlsx or .json file in a new backend/storage/inputs/ directory labeled with the Task ID (e.g., input_task123.json). - When the Python script finishes, the Queue Manager will take the final resulting JSON data and save it straight to backend/storage/outputs/output_task123.json. - The History UI will have "Download" buttons that map to endpoints returning these specific files. * GCP Cloud Storage (Scaling Up): - Local disk is great for v1, but if the container restarts (especially in Docker/Cloud Run without mounted volumes), local files are lost. - If your company allows it, integrating a GCP Service Account to save these files directly to a Google Cloud Storage (GCS) Bucket would be much more robust for production. We can design the Queue Manager to use local storage first, and swap it for GCP storage later via configuration if needed.

3. The History View (Audit & Results) * Backend: Store task metadata in tasks.json containing: taskId, scriptName, startTime, endTime, tenant, totalRecords, passCount, failCount, status (Queued, In Progress, Completed, Failed), inputFilePath, and outputFilePath. * Frontend: Create a completely new "History" page (e.g., history.html). * This page will call a new /api/scripts/history endpoint to get the list of runs and display them in a data table showing: * Script Name & Tenant * Start Time * Status Indicator (🟡 Queued, 🔵 Running, 🟢 Completed, 🔴 Failed) * Metrics: Total Records / Pass / Fail * Action columns in the table: * Download Input: Calls backend to download the stored original input data. * Download Output: Calls backend to serve the resulting output data file (if completed/failed).

3. The Execution Flow (Polling) * When a user submit a script via the UI, the frontend receives a Task ID immediately. * The frontend uses setInterval to "poll" (ask the server) every 3 seconds: "Is Task ID X done?". * The UI can show: "You are #2 in the queue..." -> "Running (12/100 processed)..." -> "Done! Here are your results."

Recommendation

Given that scripts in this platform can sometimes take a long time to run (Data Generation), Option 2 is the industry standard. However, if you are looking for a quick fix right now to stop runaway processes from dragging down the server, Option 1 is highly recommended as a first step.
